## Design Context

### Users
Solo user (developer/owner) building and using the app for personal grocery shopping. Primary
workflows: adding items when planning or noticing something running low, then checking items off
in-store. Future possibility of multi-user shared lists, but not a current concern.

### Brand Personality
Not yet defined — intentionally deferred. The app currently uses PrimeVue's Aura preset as a
neutral, modern base. Visual identity will evolve once the product scope solidifies through real
usage. Until then, the priority is a clean, functional interface that doesn't call attention to
itself.

### Aesthetic Direction
- **Current tone**: Neutral, modern, utilitarian — the Aura default
- **Anti-reference**: Anything that looks "vibe coded" — gradient-heavy hero sections, glassmorphism
  cards, excessive animations, oversaturated accent colors, trendy layouts that prioritize visual
  flair over function. The interface should look like it was built by someone who cares about the
  craft, not generated.
- **Theme**: Light and dark mode (system preference), no custom palette beyond Aura defaults for now
- **Direction**: When the time comes to develop a visual identity, it should emerge from the product's
  actual needs, not be imposed upfront

### Design Principles

1. **Foundation over decoration** — Invest in reusable tokens, consistent spacing, and well-structured
   components. Visual polish comes later; a sound system comes first.

2. **Consistent through convention** — Use the established spacing scale, PrimeVue design tokens for
   color, and scoped styles in every component. Consistency is achieved by following the system, not
   by adding more rules.

3. **Functional simplicity** — Every element should serve the task (add items, check them off). Avoid
   decorative flourishes, gratuitous transitions, or layout complexity that doesn't improve usability.

4. **Intentional, not generated** — Design choices should be deliberate and minimal. No effect or
   pattern should be added "because it looks cool." If a default works, keep the default.

5. **Accessible by default** — Follow standard accessibility practices (semantic HTML, sufficient
   contrast, keyboard navigation, touch targets) as a baseline, not as an afterthought. Aim for
   WCAG 2.1 AA compliance without over-engineering.
